## 目标
1. 把 AI 润色活动描述调整为「正式、专业」风格，把条款 / 内容要求写清楚，避免到时和探店者扯皮。
2. 在公开报名页加一个「活动参与确认协议」的勾选框,勾选才能提交,未勾选时报错提醒。
3. 协议内容根据当前活动的描述 / 名称 / 优惠券规则动态生成,涉及小红书探店时强调真实、认真撰写文案的义务,且专业无漏洞。

## 1. 重写 AI 润色 Prompt
文件：`supabase/functions/polish-activity-description/index.ts`

- 改写 `systemPrompt` / `userPrompt`：风格从"轻松口语"切回"正式、专业、面向年轻顾客但克制"。
- 强制输出 3 段结构，每条要点单独一行,中间空行:

  ```
  【活动内容】
  - …

  【内容要求】
  - …（针对小红书探店:平台、最低字数/图片数、是否带门店定位、是否需要带话题/@账号、发布时间窗口、不可删稿/隐藏的时长等)

  【活动规则】
  - …(报名资格、优惠券使用方式、一人一次、最终解释权、违规处理等)
  ```

- 规则更新:
  - 总长度放宽到 120-360 字,保证条款写得下。
  - 禁止 emoji / markdown / 通稿夸张词("惊喜钜献"等),称呼用"您"。
  - 草稿没说的具体优惠/赠品/数字不许编;但**通用条款**(最终解释权、违规取消资格、发布后不得 24h/7 天内删除等)可基于行业惯例补全。
  - 若草稿提到"小红书 / 探店 / 笔记 / 种草",必须输出「内容要求」段;否则该段可省略,只留【活动内容】+【活动规则】。

- 不动 token/鉴权/CORS/错误处理逻辑。

### 动态协议内容生成(纯前端,无需后端)
新增工具函数(同文件内或 `src/lib/voucher.ts`,放同文件更内聚):

```ts
function buildAgreement(activity, voucher): string
```

入参拼出协议正文,包括以下固定 + 动态段落:

1. **协议双方与适用范围** — 甲方:BOOMER-OFF 中古门店;乙方:报名人(您)。说明本协议适用于「{activity.name}」。
2. **活动信息** — 动态注入 `activity.name`、活动时间(`starts_at` ~ `ends_at`)、关联优惠券规则(`formatVoucherRule(voucher)` + 有效期 `valid_days` 天 + 仅到店核销)。
3. **报名信息真实性** — 您承诺姓名、手机号、上传截图等所有信息真实有效;伪造、盗用他人信息的,门店有权拒绝核销并保留追究权利。
4. **内容发布义务(探店类活动专用,基于 description 动态)** — 检测 `activity.description` 中是否包含「小红书 / 探店 / 笔记 / 种草 / 发布」关键词;命中则追加:
   - 您应在领取优惠券后 7 个自然日内,在小红书发布一条不少于 100 字、含 ≥3 张到店实拍图的真实探店笔记,并 @ 门店账号或带门店定位 / 指定话题(如活动描述另有规定的,以描述为准)。
   - 内容须基于真实到店体验,不得抄袭、搬运、AI 一键生成、虚假摆拍或夸大宣传;不得包含违法、违规、低俗信息。
   - 笔记发布后 30 天内不得删除、设为私密或大幅修改;如平台限流/审核未过,应在 3 天内重新发布。
   - 门店有权要求您提供笔记链接以核验,未按约定发布或发布后删除/隐藏的,门店有权停用未使用的优惠券、要求退还已核销的等额商品价款。
5. **优惠券使用规则** — 一人限领一次;仅到店核销;不可转让/兑现/找零;若活动描述与本条不一致,以活动描述为准。
6. **个人信息处理** — 仅用于本次活动核验、优惠券发放与必要联系,不会用于其他商业用途;保存期限不超过活动结束后 90 天。依据《个人信息保护法》,您有权要求查询、更正、删除。
7. **违约与争议处理** — 违反上述任一条款的,门店有权取消报名资格并停用相关优惠券;争议双方友好协商解决。
8. **生效** — 您勾选「我已阅读并同意」并提交报名,即视为本协议成立并对双方生效。
9. **最终解释权** — 在不违反法律强制性规定的前提下,本活动及协议的最终解释权归 BOOMER-OFF 中古门店所有。

> 触发探店段落的关键词集合先用「小红书、探店、笔记、种草、发布、文案、笔记发布、xhs」。
